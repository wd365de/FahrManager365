## Project Overview

**FahrManager 360** — a driving school management web application (Fahrschule) built with FastAPI + Jinja2 + SQLAlchemy. The UI is in German. Locally it uses SQLite; in production it uses PostgreSQL via Docker Compose.

## Commands

### Local Development (Windows PowerShell)

```powershell
# Create and activate venv
python -m venv .venv
.\.venv\Scripts\Activate.ps1

# Install dependencies
pip install -r requirements.txt

# Run development server (auto-reload)
uvicorn app.main:app --reload
```

App runs at `http://127.0.0.1:8000`. Demo credentials:
- Admin: `admin@fahrmanager360.local` / `admin123`
- Teacher: `teacher.demo@fahrmanager360.local` / `demo123`
- Students: `student1.demo@fahrmanager360.local` / `demo123`

There are no automated tests or a linter configured in this project.

### Production Deploy (VPS)

```powershell
# Full automated release (commit + push + remote deploy)
powershell -ExecutionPolicy Bypass -File .\release.ps1 -CommitMessage "release: <description>"
```

**Important:** Always run `docker compose build` on deploy — the app image is built from the local Dockerfile, so skipping the build leaves the old image running.

## Architecture

### Request Flow

```
HTTP Request → SessionMiddleware → FastAPI Router → Route handler
                                                  → Jinja2 Template (HTML response)
                                                  → RedirectResponse
```

Authentication is session-based (Starlette `SessionMiddleware` + `itsdangerous`). The session stores `user_id`. Every protected route calls `get_authenticated_user(request, db)` from `app/routes/utils.py`, then checks `user.role`.

### Routers (`app/routes/`)

| File | Prefix area | Who can access |
|------|-------------|---------------|
| `auth_routes.py` | `/login`, `/logout` | Public |
| `admin_routes.py` | `/dashboard`, `/students`, `/teachers`, `/slots`, `/settings`, `/planner` | Admin only |
| `portal_routes.py` | `/portal` | Students only |
| `appointments_routes.py` | `/appointments` | Admin + students |
| `push_routes.py` | `/push` | Authenticated users |

### Data Model (`app/models.py`)

Core hierarchy: `User` → `Student` / `Teacher` → `Appointment`, `AvailabilityWindow`

- **User**: holds `role` (`admin`, `teacher`, `student`) and credentials.
- **Student**: linked 1:1 to User; has `teacher_id` FK assigning them to a fixed teacher.
- **Teacher**: linked 1:1 to User; owns `AvailabilityWindow` records.
- **AvailabilityWindow**: a time block a teacher is available. Has `bookable_from` controlling when students can see/book it and `source` (`manual` or `planner`).
- **Appointment**: a booking by a student within a window. Statuses: `booked`, `cancelled`, `done`. Has `requires_teacher_confirmation` for request-mode bookings beyond the direct booking window.
- **AppSetting**: key/value table for planner settings (see `app/planner_settings.py`).
- **PushSubscription**: Web Push (VAPID) subscription records per user.

### Booking Logic (`app/routes/utils.py` → `build_booking_options`)

Students can book within an `AvailabilityWindow` in two modes:
- **Direct booking** (`booking_mode = "book"`): appointment starts within `STUDENT_DIRECT_BOOKING_START_LEAD_HOURS` (48h) from now.
- **Request booking** (`booking_mode = "request"`): beyond 48h but within the next `STUDENT_DIRECT_BOOKING_WINDOW_HOURS` (72h) after that. Requires teacher confirmation.

Slots are generated by sliding a cursor through the window in `BOOKING_STEP_MINUTES` (15 min) increments, checking for overlaps including a `BOOKING_BUFFER_MINUTES` (10 min) buffer.

### Schema Migrations

There is no Alembic. `run_local_schema_migrations()` in `app/main.py` runs on startup and adds missing columns via raw `ALTER TABLE` statements. When adding new columns to a model, add a corresponding migration block there.

### Configuration / Settings

- **`app/settings.py`**: Python-level constants (booking durations, buffer times, master data keys).
- **`app/planner_settings.py`**: DB-backed settings stored in `app_settings` table, managed via the admin settings page. Definitions (labels, defaults) are in `PLANNER_SETTING_DEFINITIONS` in `app/settings.py`.
- **Environment variables** (set via `.env` in production):
  - `DATABASE_URL` — defaults to local SQLite `fahrmanager360.db`
  - `SESSION_SECRET` — required in production
  - `VAPID_PUBLIC_KEY`, `VAPID_PRIVATE_KEY`, `VAPID_CLAIMS_SUB` — for Web Push notifications (optional; push UI is hidden if not configured)

### Templates

Jinja2 templates live in `app/templates/`. `base.html` is the layout template. All route handlers instantiate their own `Jinja2Templates` pointed at the same `app/templates/` directory.

### Web Push Notifications

`app/push_notifications.py` wraps `pywebpush`. Push is only active when VAPID keys are set. The service worker (`app/static/sw.js`) is served at `/sw.js` (special route to serve it at root scope for full-origin PWA registration).
